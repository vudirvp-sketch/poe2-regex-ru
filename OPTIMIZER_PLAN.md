## 0. Контекст и терминология

### 0.1 Что делает проект сейчас

Проект генерирует регулярные выражения для поиска модификаторов предметов в Path of Exile 2 (PoE2). Пайплайн:

```
fetch-poe2db.ts → normalize.ts → compute-regex.ts → compute-optimizations.ts → optimizer.ts → compiler.ts → "регекс строка"
```

- **compute-regex.ts** — для каждого токена (модификатора) вычисляет минимальный уникальный подстроку. Работает по шаблонам: берёт текст после последнего `##`/`#` (суффикс шаблона), укорачивает слева пока уникально. Если суффикс не уникален — удлиняет (Strategy 1b), берёт вторую стату (1c), или падает обратно на substring search (Strategy 2).
- **compute-optimizations.ts** — группирует токены по familyKey (нормализованный шаблон), находит общий регекс для семьи, ищет cross-family общие подстроки через LCS.
- **optimizer.ts (runtime)** — дедуплицирует одинаковые регексы в OR-группах, применяет таблицу оптимизаций (заменяет набор LITERAL на один общий).
- **regex-oracle.ts** — валидирует регекс против набора целевых и исключаемых текстов, проверяет лимит 250 символов.

### 0.2 Чего НЕ хватает — ФАКТОРИЗАЦИЯ

Текущий оптимизатор работает **по токенам**: каждый токен получает свой регекс, потом одинаковые схлопываются. Но он НЕ умеет:

1. **Факторизовать общие части между группами модов.** Пример:
   - «сопротивлению огню» (20 символов)
   - «сопротивлению холоду» (22 символа)
   - «сопротивлению молнии» (22 символа)
   - Flat OR: `сопротивлению огню|сопротивлению холоду|сопротивлению молнии` = 64 символа
   - Факторизованный: `сопротивлению (огню|холоду|молнии)` = 34 символа — **экономия 47%**

2. **Использовать Trie (префиксное дерево) для оптимальной группировки.**

3. **Применять динамическое программирование на Trie** для выбора оптимальной факторизации.

4. **Учитывать диалект PoE2** при оптимизации — `.`, `[]`, `?`, `.*`.

### 0.3 Диалект PoE2 regex — проверенные в игре факты

| Фича | Поведение | Влияние на оптимизатор |
|------|-----------|----------------------|
| `.` | Матчит **ЛЮБОЙ** символ | Использовать с осторожностью — Oracle проверяет FP |
| `[0-9]` | Матчит только цифру | Теперь используется вместо `.` в числовых паттернах (Фаза 1 выполнена) |
| `!` | NOT — скрывает предметы | — |
| `.*` | Пересекает границы модов | prefix anchoring критичен |
| `\|` | OR — стандартное поведение | — |
| `()` | Группировка | Для факторизации |
| `[]` | Класс символов | `[её]` для ёфикации, `[юя]` для окончаний |
| `?` | Опциональность | Полезно для опциональных окончаний |
| Регистр | Case-insensitive поиск | Все регексы в нижнем регистре |
| Лимит | **250 символов** на один регекс | Оптимизатор должен следить за лимитом |
