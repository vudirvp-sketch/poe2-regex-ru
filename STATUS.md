## Текущая итерация: 35 — Глубокий анализ: `|` не работает с многословными паттернами

### Краткое резюме

Обнаружены **ТРИ взаимосвязанных бага**, корневая причина — PoE2 движок не поддерживает `|`
между многословными (содержащими пробелы) альтернативами. Это ломает OR-режим,
оптимизационную таблицу и regexExclude.

### Баг A: `|` не работает с многословными паттернами (CRITICAL)

**Подтверждено тестами 9, 10, 11:** PoE2 токенизирует поисковую строку по пробелам.
`|` OR-ит только соседние слова до следующего пробела.

| Ввод | Результат | Вывод |
|------|-----------|-------|
| `скорости атаки\|перезарядки умений` | Ничего | `\|` с пробелами = сломано |
| `скорости копьями` | AND-поведение | Пробел = AND |
| `скорости атаки(?!.*луками)` | Работает | `(?!…)` — per-block |

**Последствие:** Оптимизационная таблица генерирует паттерны вида
`повышение (брони\|скорости (атаки\|сотворения чар)\|шанса)` — все `\|` между
многословными альтернативами **не работают** в PoE2.

**Масштаб проблемы:**

| Категория | Opt-записей с многословным `\|` | Всего opt-записей |
|-----------|-------------------------------|-------------------|
| Самоцветы | ~30 | 113 |
| Амулеты | 53 | 114 |
| Кольца | 44 | 93 |
| Пояса | 41 | 83 |
| Плиты | 26 | 35 |

**КРИТИЧЕСКИЙ НЕИЗВЕСТНЫЙ ВОПРОС:** Работает ли `|` внутри `()` с многословными альтернативами?
Тест 9 проверял только TOP-LEVEL `|`. Нужно проверить `(скорости атаки|передвижения)`.

Если `()` сохраняет `|` для многословных — баг только в компиляторе (Bug B).
Если нет — оптимизационная таблица **фундаментально сломана** для всех категорий.

### Баг B: AND-in-OR = вложенные кавычки (CRITICAL)

`compiler.ts` compileInner(AND) оборачивает children в `"..."`, даже внутри OR.
PoE2 не поддерживает вложенные кавычки — OR-структура разрушается.

**Было:** `"A" "!X|Y"|B|C` (вложенные кавычки)
**Предлагалось:** `(A !X|Y)|B|C` → `"(A !X|Y)|B|C"`

**⚠️ НО:** Если `|` не работает с многословными даже внутри `()`, то `(A !X|Y)` тоже
не будет работать корректно, если A — многословный паттерн. Решение Bug B зависит
от результата тестирования Bug A.

### Баг C: regexExclude — неполные формы слов (MEDIUM)

Текущие exclude-паттерны используют конкретные словоформы:
- `самострелами` (творительный) → НЕ ловит `самострела` (родительный: «перезарядки самострела»)
- `посохами` → НЕ ловит `посоха`
- Нужно использовать усечённые основы: `самострел`, `посох`, `копь`

**Результаты Тестов 15-17 определят путь исправления:**

### Путь A: Если `()` + `|` + многословные РАБОТАЕТ

Оптимизационная таблица корректна. Нужно исправить только компилятор:

| Шаг | Содержание | Затронуто |
|-----|-----------|-----------|
| A1 | Компилятор: `inOr` контекст + `()` для AND-in-OR | `compiler.ts` |
| A2 | Оптимизатор: обновить `applyOptimizationTable` / `removeConflictingExcludes` | `optimization-strategies.ts`, `core-optimizations.ts` |
| A3 | regexExclude: усечённые основы слов | ETL `compute-regex-strategies.ts` |
| A4 | In-game верификация: самоцветы | — |
| A5 | Распространение: амулеты, кольца, пояса, плиты | — |

### Путь B: Если `()` + `|` + многословные НЕ РАБОТАЕТ

Оптимизационная таблица **фундаментально сломана**. Нужен полный редизайн:

| Шаг | Содержание | Затронуто |
|-----|-----------|-----------|
| B1 | ETL: заменить многословные `\|` в opt-таблице на AND-комбинации | `compute-optimizations.ts` |
| B2 | Компилятор: для OR-режима генерировать `"A" "B"\|"C" "D"` (AND внутри OR) | `compiler.ts` |
| B3 | Runtime: каждый токен = отдельная quoted group, OR через внешние `"..."` группы | `useCategoryPage.ts` |
| B4 | regexExclude: переработать на `(?!…)` per-block exclusions | ETL + compiler |
| B5 | In-game верификация | — |
| B6 | Распространение на все категории | — |

**Альтернатива для Пути B:** Использовать `(?!…)` per-block подход:
- Общее слово (напр. «скорости») + `(?!.*исключение1)(?!.*исключение2)` для каждого блока
- Плюс: per-block точность, не нужна `\|` с многословными
- Минус: длинные регексы, нужно покрытие всех словоформ
