### P3: Префиксное закрепление — `^` anchor проверка в игре
- Phase 9a подтвердила: enumeration НЕ решает range notation FP полностью
- Числа из range notation (напр. "27" из "(27-50)") совпадают с enumerated значениями
- Возможное решение: `^` anchor (`"^(2[7-9]|30).*откладывания наград"`)
- Нужна проверка в игре: работает ли `^` для начала строки мода в PoE2?
- Альтернатива: suffix anchoring (`"(2[7-9]|30)%.*откладывания наград"`) — `%` после числа, перед range notation

### Баг-профиль для будущего аудита
1. **Range notation FP (Phase 9a, CONFIRMED)** — enumeration `(2[7-9]|30).*suffix` подсвечивает 26% и 22% предметы. Числа из range notation в тексте предмета (напр. "(27-50)") совпадают с enumerated значениями. Возможное решение: `^` anchor (нужна проверка в игре).
2. **Edge cases с OR-suffix RANGE** — при разных (min,max) но одинаковых суффиксах, объединение может быть некорректным
3. **VendorPage numeric-only без чекбокса** — свойство с numericInput, но без чекбокса (selectedIds), может не попасть в regex
