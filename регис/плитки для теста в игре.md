# Плитки для in-game тестирования регекса

> Тест-кейсы для проверки корректности генерируемых regex в PoE2 (RU клиент).
> Каждый тест = конкретный предмет из инвентаря + regex для вставки в поиск + ожидаемый результат.

---

## Как читать тесты

Формат каждого теста:
```
T##  [КАТЕГОРИЯ]  Паттерн: <описание проверяемого паттерна>
Предмет: <название предмета из инвентаря>
Regex: "<сгенерированный regex>"
Ожидание: <подсветить / НЕ подсвечивать / частично>
Проверка: <что именно проверяем>
```

**Предметы** — ниже в секции «Список предметов». Бери их из инвентаря/стэша.

## Группа A: Want + Single Exclude (базовый PN-паттерн)

Проверяет: `"want" "!dontwant"` — один want + один exclude.

### A1 [RING] Want + exclude другой стихии
Предмет: Отвратительное потрясение (Кольцо Разлома)
Regex: `"к сопротивлению молнии" "!к сопротивлению огню"`
Ожидание: Подсвечивает (предмет имеет молнию, но НЕ имеет огонь на этом предмете — проверь, есть ли огонь)
Проверка: PN-паттерн `!` внутри кавычек, exclude по стихии

### A2 [RING] Want + exclude стихии, которая ЕСТЬ
Предмет: Отвратительное потрясение (Кольцо Разлома)
Regex: `"к сопротивлению молнии" "!уклонению"`
Ожидание: НЕ подсвечивает (предмет имеет +121 к уклонению — exclude срабатывает item-wide)
Проверка: `!X` — item-wide, исключает весь предмет если X в ЛЮБОМ блоке

### A3 [AMULET] Want + exclude с regexExclude (к силе)
Предмет: Унылый фермуар (Лазурный амулет)
Regex: `"к силе" "!к силе и"`
Ожидание: Подсвечивает (на предмете есть +33 к интеллекту, а не «к силе и ловкости»)
Проверка: regexExclude предотвращает FP на модах типа «к силе и ловкости»

### A4 [AMULET] Want + exclude «Приспеш» (breachborn-разделение)
Предмет: Унылый фермуар (Лазурный амулет) — имеет +12% к сопротивлению холоду
Regex: `"к сопротивлению всем стихиям" "!Приспеш"`
Ожидание: Подсвечивает только если на предмете есть «к сопротивлению всем стихиям» без «Приспешники»
Проверка: exclude «Приспеш» отсекает breachborn-моды «Приспешники имеют +...% к сопротивлению всем стихиям»

## Группа B: Want + Multiple Exclude

Проверяет: `"want" "!excl1|excl2"` — один want + OR-exclude нескольких паттернов.

### B1 [AMULET] Want «повышение шанса критического удара» + exclude «Приспеш|для»
Предмет: Унылый фермуар (Лазурный амулет) — имеет 15% повышение шанса критического удара
Regex: `"повышение шанса критического удара" "!Приспеш|для"`
Ожидание: Подсвечивает (мод без «Приспеш» и без «для»)
Проверка: Множественный exclude через `|` внутри `!` — `"!A|B"` исключает предмет если есть A ИЛИ B

### B2 [AMULET] Want «увеличение урона от чар» + exclude «при»
Предмет: Унылый фермуар (Лазурный амулет)
Regex: `"увеличение урона от чар" "!при"`
Ожидание: Предмет НЕ имеет «увеличение урона от чар» → не подсвечивает
Проверка: exclude «при» отсекает breachborn-мод «увеличение урона от чар при...» (во время действия флакона)

### B3 [RING] Want «увеличение урона» + exclude «увеличение урона от|во время|флакона|чар»
Предмет: Отвратительное потрясение (Кольцо Разлома)
Regex: `"увеличение урона" "!увеличение урона от|во время|флакона|чар"`
Ожидание: НЕ подсвечивает (нет мода «увеличение урона» самостоятельного)
Проверка: Множественный exclude с длинными паттернами — budget-aware

### B4 [AMULET] Want «увеличение уклонения» + exclude «вашего»
Предмет: Унылый фермуар (Лазурный амулет) — имеет 43% увеличение уклонения
Regex: `"увеличение уклонения" "!вашего"`
Ожидание: Подсвечивает (мод «увеличение уклонения», не «увеличение вашего уклонения»)
Проверка: exclude «вашего» отсекает другую семью модов

## Группа C: OR Mode + Exclude

Проверяет: OR-mode + exclude — все want-моды идут в один OR, exclude отдельно.

### C1 [RING] OR mode: молния ИЛИ огонь + exclude хаос
Предмет: Отвратительное потрясение (Кольцо Разлома) — имеет 28% увеличение урона от огня + +35% к сопротивлению молнии
Regex: `"к сопротивлению огню|к сопротивлению молнии" "!хаос"`
Ожидание: Подсвечивает (есть огонь или молния, нет хаоса)
Проверка: OR внутри кавычек + exclude

### C2 [RING] OR mode: молния ИЛИ огонь + exclude холоду
Предмет: Ненавистное потрясение (Кольцо Разлома) — имеет 15% увеличение урона хаосом, НЕТ сопротивления огню/молнии/холоду
Regex: `"к сопротивлению огню|к сопротивлению молнии" "!холоду"`
Ожидание: НЕ подсвечивает (нет ни огня, ни молнии)
Проверка: OR не находит — предмет отсекается, даже если exclude тоже не найден

### C3 [AMULET] OR mode: «к силе|к ловкости» + exclude «к интеллекту»
Предмет: Унылый фермуар (Лазурный амулет) — +33 к интеллекту, НЕТ силы/ловкости
Regex: `"к силе|к ловкости" "!к интеллекту"`
Ожидание: НЕ подсвечивает (нет силы/ловкости, даже хотя exclude не срабатывает — нет «к интеллекту» в виде отдельного мода... хотя на предмете есть +33 к интеллекту)
Проверка: AND-логика — want должен быть true, exclude может быть false, но если нет want — не подсвечивает

### C4 [AMULET] OR mode: «к силе|к ловкости|к интеллекту» + exclude «уклонению»
Предмет: Унылый фермуар (Лазурный амулет) — +33 к интеллекту, 43% увеличение уклонения
Regex: `"к силе|к ловкости|к интеллекту" "!уклонению"`
Ожидание: НЕ подсвечивает (есть «к интеллекту», но exclude «уклонению» срабатывает item-wide)
Проверка: OR-match на want + item-wide exclude

## Группа D: Ranged Exclude с конкретными значениями

Проверяет: exclude для ranged-модов — `"want" "!(number-pattern)"` — exclude определённого числового диапазона.

### D1 [WAYSTONE] Want путевые камни ≥85% + exclude «отравление»
Предмет: Призрачный камень (Путевой камень Ур. 15) — Шанс выпадения +85%, есть «отравление»
Regex: `"Шанс выпадения" "([89][0-9]|[1-9][0-9][0-9])%" "!отравлен"`
Ожидание: НЕ подсвечивает (exclude «отравлен» срабатывает на мод «Монстры ... могут наложить отравление»)
Проверка: Ranged want + текстовый exclude — exclude работает item-wide

### D2 [WAYSTONE] Want путевые камни ≥85% + exclude «критического»
Предмет: Призрачный камень (Путевой камень Ур. 15) — имеет повышение шанса критического удара
Regex: `"Шанс выпадения" "([89][0-9]|[1-9][0-9][0-9])%" "!критического"`
Ожидание: НЕ подсвечивает (exclude «критического» срабатывает на мод «повышение шанса критического удара»)
Проверка: Ranged want + exclude по ключевому слову из другого мода

### D3 [RING] Want +39% к сопротивлению молнии (ranged) + exclude «огню»
Предмет: Отвратительное потрясение (Кольцо Разлома) — +35% к сопротивлению молнии, 28% урон от огня
Regex: `"(3[0-9]%|4[0-9]%).*к сопротивлению молнии" "!огню"`
Ожидание: НЕ подсвечивает (есть «огню» в моде «увеличение урона от огня» — exclude срабатывает)
Проверка: Ranged exclude по другому семейству модов

### D4 [TABLET] Want ≥8 зарядов + exclude «золот»
Предмет: Потусторонний ордер (Заражённая плитка) — 10 зарядов, НЕТ золота
Regex: `"зарядов.*([89]|[1-9][0-9])" "!золот"`
Ожидание: Подсвечивает (10 зарядов ≥8, нет «золот»)
Проверка: Ranged want на implicit + exclude по тексту

### D5 [TABLET] Want ≥8 зарядов + exclude «Ритуала»
Предмет: Потусторонний ордер (Заражённая плитка) — implicit «Добавляет Заражение», НЕ «Ритуала»
Regex: `"зарядов.*([89]|[1-9][0-9])" "!Ритуала"`
Ожидание: Подсвечивает (нет «Ритуала» в тексте предмета)
Проверка: Ranged want + exclude по implicit-типу

## Группа E: regexPrefixContext (AND-композиция через контекст)

Проверяет: `"context" "regex"` — AND двух quoted-групп, где context — уникальное слово семейства.

### E1 [RING] breachborn: «имеют» + «увеличение урона»
Предмет: Нужен предмет с breachborn-модом «Приспешники имеют ... увеличение урона»
Regex: `"имеют" "увеличение урона"`
Ожидание: Подсвечивает breachborn-мод, НЕ подсвечивает обычный «увеличение урона от атак»
Проверка: regexPrefixContext отсекает FP — обычные моды не содержат «имеют»

### E2 [RING] breachborn: «имеют» + «стихия»
Предмет: Нужен предмет с breachborn-модом «Приспешники имеют +...% к сопротивлению всем стихиям»
Regex: `"имеют" "стихия"`
Ожидание: Подсвечивает breachborn-мод, НЕ подсвечивает обычные «к сопротивлению всем стихиям»
Проверка: context отсекает FP

### E3 [AMULET] breachborn: «имеют» + «повышение шанса критического удара»
Предмет: Нужен предмет с breachborn «Приспешники имеют ... повышение шанса критического удара»
Regex: `"имеют" "повышение шанса критического удара"`
Ожидание: Подсвечивает breachborn, НЕ подсвечивает обычный «повышение шанса критического удара» (нет «имеют»)
Проверка: context + exclude совместно — context для разделения семейств

### E4 [JEWEL] breachborn: «имеют» + оптимизированный OR-regex
Предмет: Нужен самоцвет с breachborn-модом (Приспешники имеют...)
Regex: `"имеют" "увеличение (области действия|максимума ...|уклонения|...)"`
Ожидание: Подсвечивает только breachborn-моды с «имеют», не обычные
Проверка: Сложный OR внутри optimization entry + context

### E5 [TABLET] context: «снова» + «быстрее»
Предмет: Непостижимая земля (Плитка Ритуала) — «Отложенные награды ... появляются снова на 40% быстрее»
Regex: `"снова" "быстрее"`
Ожидание: Подсвечивает (оба слова есть в одном блоке)
Проверка: AND across blocks для regexPrefixContext — но тут оба слова в одном блоке, поэтому .* бы тоже сработало, но AND надёжнее

## Группа F: Enumerated Range + Suffix Anchors

Проверяет: `"(27|28|29|30).*suffix"` — decade grouping, % anchor, ^ anchor.

### F1 [TABLET] Enumerated range: редкость 8-12%
Предмет: Потусторонний ордер (Заражённая плитка) — НЕТ мода «увеличение редкости» (только количество редких монстров)
Regex: `"(8|9|1[0-2])%.*карте предметов"`
Ожидание: НЕ подсвечивает (нет мода редкости на этом предмете)
Проверка: enumeration без FP

### F2 [TABLET] Enumerated range: количество редких монстров 25-35%
Предмет: Потусторонний ордер (Заражённая плитка) — 39% увеличение количества редких монстров
Regex: `"(2[5-9]|3[0-5])%.*редких монстров"`
Ожидание: НЕ подсвечивает (39% > 35%)
Проверка: enumerated range НЕ выходит за верхнюю границу

### F3 [TABLET] Enumerated range: количество редких монстров 25-40%
Предмет: Потусторонний ордер (Заражённая плитка) — 39% увеличение количества редких монстров
Regex: `"(2[5-9]|3[0-9]|40)%.*редких монстров"`
Ожидание: Подсвечивает (39% в диапазоне 25-40%)
Проверка: enumeration с decade grouping + % anchor

### F4 [WAYSTONE] ^ anchor: урон монстров ≥27%
Предмет: Призрачный камень — НЕТ мода «увеличение урона монстров» (только критический)
Regex: `"^(2[7-9]|[3-9][0-9]|[1-9][0-9][0-9])%.*увеличение урона монстров"`
Ожидание: НЕ подсвечивает
Проверка: ^ anchor — мод начинается с числа

### F5 [RING] % anchor: сопротивление ≥31%
Предмет: Отвратительное потрясение — +35% к сопротивлению молнии
Regex: `"(3[1-9]|4[0-9])%.*к сопротивлению молнии"`
Ожидание: Подсвечивает (35% ≥ 31%)
Проверка: % anchor предотвращает FP от range notation

## Группа G: Reversed Regex (Implicit Patterns)

Проверяет: `"suffix.*number%"` — текст ДО числа, для implicit-модов.

### G1 [WAYSTONE] Reversed: Шанс выпадения ≥85%
Предмет: Призрачный камень — Шанс выпадения путевого камня: +85%
Regex: `"падения.*([89][0-9]|[1-9][0-9][0-9])%"`
Ожидание: Подсвечивает (85% ≥ 85%)
Проверка: reversed regex — текст.*число% (имплисит не dual-indexed)

### G2 [WAYSTONE] Reversed: Шанс выпадения ≥100%
Предмет: Разрушенный коридор — Шанс выпадения путевого камня: +110%
Regex: `"падения.*([1-9][0-9][0-9])%"`
Ожидание: Подсвечивает (110% ≥ 100%)
Проверка: reversed для 3-digit чисел

### G3 [WAYSTONE] Reversed: Редкость ≥30%
Предмет: Изменённый прогресс — Редкость предметов: +36%
Regex: `"едкость.*([3-9][0-9]|[1-9][0-9][0-9])%"`
Ожидание: Подсвечивает (36% ≥ 30%)
Проверка: короткий regex «едкость» (truncated «Редкость»)

### G4 [TABLET] Reversed + colon anchor: дополнительных редких монстров ≥2
Предмет: Возвышенная Плитка Бездны чемпионов — «появляется дополнительных редких монстров: 1»
Regex: `"появляется.*: ([2-9]|[0-9][0-9][0-9]?)"`
Ожидание: НЕ подсвечивает (значение 1 < 2, colon anchor гарантирует что 1 после `: ` не матчит [2-9])
Проверка: colon anchor предотвращает FP от range notation

### G5 [TABLET] Reversed + colon anchor: дополнительных редких сундуков ≥3
Предмет: Тревожный суд — «дополнительных редких сундуков: 2»
Regex: `"х редких с.*: ([3-9]|[0-9][0-9][0-9]?)"`
Ожидание: НЕ подсвечивает (2 < 3, colon anchor)
Проверка: ранее обнаруженный FP (range "2(1-3)" → "3" матчил) — теперь исправлено через colon anchor

### G6 [TABLET] Implicit: зарядов ≥5
Предмет: Потусторонний ордер — Осталось зарядов - 10
Regex: `"сь зарядов.*([5-9]|[1-9][0-9])"`
Ожидание: Подсвечивает (10 ≥ 5)
Проверка: implicit reversed без % anchor (заряды — число без %)

## Группа H: Dual-Number (Prefix Anchoring)

Проверяет: `"От (number).*урона к атакам"` — prefix для dual-number модов.

### H1 [RING] Dual-number: физический урон к атакам (min ≥10)
Предмет: Отвратительное потрясение — нет мода физического урона к атакам
Regex: `"От ([1-9][0-9]).*урона к атакам"`
Ожидание: НЕ подсвечивает
Проверка: prefix «От» привязывает число к первому placeholder

### H2 [RING] Dual-number: огонь к атакам (slot 0 ≥10)
Предмет: Расколотый завиток — «Добавляет от 17(13-19) до 30(27-32) урона от огня к атакам»
Regex: `"От ([1-9][0-9]).*огня к атакам"`
Ожидание: Подсвечивает (17 ≥ 10)
Проверка: prefix «От» + suffix «огня к атакам» — правильный слот

### H3 [BELT] Dual-number: урон шипами (min ≥5)
Предмет: Нужен предмет с модом «От ## до ## физического урона шипами»
Regex: `"От ([5-9]|[1-9][0-9]).*урона шипами"`
Ожидание: Подсвечивает если min урона шипами ≥5
Проверка: dual-number prefix anchoring на поясах

## Группа I: Yofication ([её])

Проверяет: `[её]` замена в regex — корректность матчинга обоих вариантов.

### I1 [RING] Yofication: «к силе» → «к сил[её]»
Предмет: Отвратительное потрясение — +23 к силе
Regex: `"к сил[её]"`
Ожидание: Подсвечивает
Проверка: `[её]` матчит «к силе» (в игре текст содержит «к силе»)

### I2 [AMULET] Yofication: «к ловкости» → «к ловкост[её]»
Предмет: Крутящий горжет — +14 к ловкости
Regex: `"к ловкост[её]"`
Ожидание: Подсвечивает
Проверка: `[её]` в другом контексте

### I3 [WAYSTONE] Yofication: «регенерации маны»
Предмет: Любой предмет с модом «повышение скорости регенерации маны»
Regex: `"регенерац[её].*маны"`
Ожидание: Подсвечивает
Проверка: `[её]` в середине слова

## Группа J: AND Across Blocks

Проверяет: `"X" "Y"` — AND двух quoted-групп, X и Y в разных блоках предмета.

### J1 [WAYSTONE] AND: Шанс выпадения + Осквернено
Предмет: Призрачный камень — Шанс +85%, Осквернено
Regex: `"Шанс выпадения" "Осквернено"`
Ожидание: Подсвечивает
Проверка: AND across blocks — implicit-блок + state-блок

### J2 [RING] AND: здоровье + молния
Предмет: Отвратительное потрясение — +66 к максимуму здоровья + +35% к сопротивлению молнии
Regex: `"максимуму здоровья" "к сопротивлению молнии"`
Ожидание: Подсвечивает
Проверка: AND across blocks — два разных мод-блока

### J3 [AMULET] AND: регенерация маны + меткость
Предмет: Унылый фермуар — 28% повышение скорости регенерации маны + +380 к меткости
Regex: `"регенерации маны" "меткости"`
Ожидание: Подсвечивает
Проверка: AND implicit + prefix мод

### J4 [RING] AND НЕ работает если порядок обратный и в одном блоке
Предмет: Отвратительное потрясение — «+35% к сопротивлению молнии»
Regex: `"молнии.*к сопротивлению"` (обратный порядок в одном блоке)
Ожидание: НЕ подсвечивает (в блоке «к сопротивлению» стоит ДО «молнии»)
Проверка: `.*` направлен — forward only

## Группа K: Complex Combinations (Budget-Aware)

Проверяет: несколько want + несколько exclude — итоговый regex в пределах 250 символов.

### K1 [AMULET] 3 want + 2 exclude
Предмет: Унылый фермуар
Regex: `"к интеллекту" "повышение шанса критического удара" "к сопротивлению холоду" "!Приспеш|для"`
Ожидание: Подсвечивает (все 3 want есть, нет Приспеш/для)
Проверка: multi-want + multi-exclude, regex длина < 250

### K2 [RING] 4 want (OR mode) + 1 exclude
Предмет: Отвратительное потрясение
Regex: `"к силе|к сопротивлению молнии|максимуму здоровья|уклонению" "!хаосом"`
Ожидание: Подсвечивает (есть сила/молния/здоровье/уклонение, нет хаоса)
Проверка: OR mode с 4 альтернативами + exclude

### K3 [TABLET] 2 want + 2 exclude (сложный)
Предмет: Потусторонний ордер (Заражённая плитка)
Regex: `"зарядов.*([89]|[1-9][0-9])" "путевых камн" "!золот|Ритуала"`
Ожидание: Подсвечивает (10 зарядов, есть «путевых камней», нет «золот» и «Ритуала»)
Проверка: ranged want + text want + multi-exclude

## Группа L: Optimization Table — Complex OR Patterns

Проверяет: оптимизированные regex из optimizationTable с несколькими суффиксами через `|`.

### L1 [RING] Opt: «увеличение (урона( от (огня|холода|молнии|чар)| хаосом)?|бонуса к критическому урону)» + context «имеют»
Предмет: Нужен предмет с breachborn-модом (Приспешники имеют...)
Regex: `"имеют" "увеличение (урона( от (огня|холода|молнии|чар)| хаосом)?|бонуса к критическому урону)"`
Ожидание: Подсвечивает breachborn-мод
Проверка: вложенный OR внутри optimization entry + context

### L2 [JEWEL] Opt: «увеличение (области действия|максимума ...|уклонения|...)» + context «имеют»
Предмет: Нужен самоцвет с breachborn-модом
Regex: `"имеют" "увеличение (области действия|максимума (энергетического щита|здоровья)|уклонения|урона в ближнем бою|бонуса к критическому урону|количества получаемых зарядов флакона|длительности эффекта умения|порога оглушения)"`
Ожидание: Подсвечивает breachborn-мод
Проверка: длинный OR внутри optimization — budget-aware (может не поместиться в 250)

### L3 [TABLET] Opt: «можно встретить дополнительную» + exclude «без|бездну»
Предмет: Потусторонние странники (Плитка Бездны) — «можно встретить ларцы»
Regex: `" можно встретить дополнительную" "!без|бездну"`
Ожидание: НЕ подсвечивает (на предмете «можно встретить ларцы», не «дополнительную»)
Проверка: optimization entry regex + exclude

## Чего НЕ хватает в текущем наборе предметов

Текущий набор предметов покрывает: кольца (3), путевые камни (3), плитки (11), амулеты (3).

**Критически не хватает:**

### 1. Пояса (0 предметов) — ВАЖНО
Пояса имеют уникальные паттерны: dual-number «От ## до ## урона шипами» (prefix anchoring),
colon-anchored моды «Обереги получают зарядов в секунду: ##» и breachborn-моды.
**Нужно:** минимум 2 пояса — один с breachborn-модом (Приспешники имеют...), один с уроном шипами.

### 2. Самоцветы (0 предметов) — ВАЖНО
Самоцветы — самая сложная категория для regexPrefixContext + optimizationTable.
Breachborn-моды «Приспешники имеют ...» — основной источник cross-family FP.
**Нужно:** минимум 2 самоцвета — один с breachborn-модом (для E1-E5, L1-L2), один обычный/осквернённый.

### 3. Реликвии (0 предметов) — НИЗКИЙ ПРИОРИТЕТ
Реликвии не имеют regexPrefixContext и regexExclude. Простые regex без exclude/context.
Тестировать паттерны можно на имеющихся категориях. **Не обязательно**, но полезно для полноты.

### 4. Очернённые (desecrated) предметы — СРЕДНИЙ ПРИОРИТЕТ
Тесты E1-E5 и L1-L2 требуют breachborn-моды. В текущем наборе НЕТ предметов с breachborn-модами
(только обычные/нормальные). Очернённые плитки есть, но без breachborn-модов на аксессуарах.
**Нужно:** минимум 1 кольцо/амулет/пояс с breachborn-модом «Приспешники имеют...»

### Резюме дефицита
| Категория | В наборе | Нужно | Приоритет |
|-----------|----------|-------|-----------|
| Пояса | 0 | 2 | Критический |
| Самоцветы | 0 | 2 | Критический |
| Реликвии | 0 | 0-1 | Низкий |
| Очернённые (breachborn) | 0 | 1-2 | Высокий |

**Без поясов и самоцветов с breachborn-модами невозможно протестировать:**
- Группы E (regexPrefixContext) — все 5 тестов
- Группы L (optimization table OR) — все 3 теста
- Группу H (dual-number prefix) — тест H3

**Существующий набор предметов ДОСТАТОЧЕН для:**
- Группы A-D (want + exclude, ranged exclude) — ✅
- Группы F (enumerated range + anchors) — ✅
- Группы G (reversed regex) — ✅
- Группы I (yofication) — ✅
- Группы J (AND across blocks) — ✅
- Группы K (complex combinations) — ✅ (частично, без breachborn)

## Приоритет тестирования

1. **Сначала** группы A, B, D — базовый want+exclude (самые частые юзкейсы)
2. **Затем** группы G, F — reversed regex и enumerated ranges (уникальные для PoE2)
3. **Затем** группы C, J — OR mode и AND across blocks
4. **Затем** группа E — regexPrefixContext (нужны breachborn предметы!)
5. **Затем** группы H, I, K, L — продвинутые паттерны
