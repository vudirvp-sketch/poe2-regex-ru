## 6. Key Architecture Points

### PoE2 Regex Dialect (NOT standard PCRE)

Documented in detail in `docs/ARCHITECTURE.md` ¬ß3. Key rules:

| Syntax | Meaning | Verified |
|--------|---------|----------|
| `substring` | Simple substring match | ‚úÖ |
| `\|` | OR (alternation) | ‚úÖ |
| `!` | NOT (negation) | ‚úÖ |
| `""` | Phrase grouping + AND separator | ‚úÖ |
| `.` | Any single character | ‚úÖ |
| `.*` | Any sequence WITHIN a single block | ‚úÖ |
| `[]` | Character class | ‚úÖ |
| `^` | Start-of-block anchor | ‚úÖ |
| `()` | Grouping | ‚úÖ |
| `\d` | Digit shorthand | ‚úÖ |

**NOT supported:** `?` (optional), `.*` across blocks, negative lookahead, non-greedy quantifiers, backreferences. `$` end anchor is unreliable ‚Äî do not use.

**Critical rules:**
1. `!` must be INSIDE quotes when combined with `|`: `"!A|B"` works, `!"A|B"` does NOT
2. `.*` does NOT cross block boundaries ‚Äî each mod/implicit/property is a separate block
3. `.*` is directional ‚Äî forward only within a block
4. AND via space between quoted groups is order-independent and works ACROSS blocks
5. `!X` is item-wide ‚Äî excludes entire item if X appears in ANY block
6. Case insensitive ‚Äî verified with Cyrillic

### Block-Based Matching Model

Each piece of item text is an independent searchable block. `.*` works ONLY within a single block. AND (`"X" "Y"`) works ACROSS blocks.

### 3-Level FP Prevention

| Level | Method | When | FP prevented |
|-------|--------|------|-------------|
| 1 | `^` (anchorStart) | Template starts with `##` | Range notation at non-position-0 |
| 2 | `%` suffix anchor (anchorEnd) | Template has `##%` AND anchorStart=false | Numbers not followed by `%` |
| 3 | Enumeration (compact decade) | Range ‚â§ 50 values | Secondary numbers not matching enumerated values |

### Compiler: Enumerated Range + AND Fallback

- **Enumerated** (‚â§50 values): single quoted group with compact decade grouping ‚Üí `"(2[7-9]|30).*suffix"`
- **AND Fallback** (>50 values): two AND-joined groups ‚Üí `"‚â•min.*suffix" "‚â§max.*suffix"`

### Positive + Negative Mods (Want + Don't-Want)

Pattern: `"want1|want2" "!dontwant1|dontwant2"`

Architecture: `excludeMode=false` ‚Üí positive matches, `excludeMode=true` ‚Üí EXCLUDE(OR(...)). Combined in `buildAstFromSelections()`.

### 250-Char Budget

Budget estimation in `limits.ts`. Optimization layers: family grouping, DP factorization, suffix shortening, yofication, optimization table.

## 7. ETL Pipeline Overview

```
poe2db.tw/ru ‚Üí fetch ‚Üí normalize ‚Üí filter implicits ‚Üí compute regex
‚Üí compute optimizations ‚Üí generate JSON ‚Üí i18n overrides ‚Üí FP repair
‚Üí patch opt entries ‚Üí iterative optimizer ‚Üí public/generated/*.json
```

Full details in `docs/ETL_GUIDE.md`.

**i18n overrides:** `scripts/etl/i18n-overrides.json` ‚Äî manually verified Russian translations for tokens that poe2db.tw shows in English only (~56 overrides).

**Source hash:** Each generated JSON contains `sourceHash` ‚Äî SHA-256 of source HTML. `pnpm etl:check-stale` compares against cache to detect data changes.

## 11. Frequent Pitfalls

1. **`!` must be INSIDE quotes with `|`:** `"!A|B"` works, `!"A|B"` does NOT
2. **`.*` does NOT cross block boundaries** ‚Äî use AND (`"X" "Y"`) for cross-block search
3. **`$` end anchor is unreliable** ‚Äî do not use
4. **Implicit-set bonuses are NOT searchable** ‚Äî filtered out in ETL Step 2b
5. **Core must stay dependency-free** ‚Äî never add npm imports to `src/core/`
6. **Generated JSON is read-only** ‚Äî never edit `public/generated/` by hand
7. **Types in `types.ts` ONLY** ‚Äî no duplicate interface definitions elsewhere
8. **Vendor properties in `vendor-properties.ts` ONLY** ‚Äî canonical source
9. **`round10` always disabled for enumerated ranges** ‚Äî enumeration is inherently precise
10. **`reversed=true` for implicit tokens** ‚Äî implicit blocks have number AFTER text, regex pattern is `"suffix.*(number)%"`
11. **Colon anchor (`: `)** ‚Äî for non-% reversed mods ending with `: ##`, prevents FP from range notation
12. **Word truncation works at END only** ‚Äî `"–∫ —Å–∏"` matches `"–∫ —Å–∏–ª–µ"`, mid-word does NOT work. Min 3 significant chars.
13. **Minimum regex length** ‚Äî optimizer enforces min 5/7/10 chars by category; shorter gets `regexPrefixContext`
14. **DIALECT_PAIRS** ‚Äî only genuine phonological alternates (–µ/—ë, —é/—è, –∞/—è, —ã/–µ, –∏/–µ). Never add –æ‚Üí–≤ or similar non-alternating pairs
15. **VendorPage uses `useCategoryPage` with `customData`** ‚Äî `buildVendorCategoryData()` converts VENDOR_PROPERTIES into CategoryData format. No separate `useVendorPage` hook. GROUP_COLORS derived from `tags: [group:${group}]`
16. **Vendor numeric properties use reversed pattern** ‚Äî `"suffix.*number"` not `"number.*suffix"` because game property lines have text before number (e.g. "–£—Ä–æ–≤–µ–Ω—å –ø—Ä–µ–¥–º–µ—Ç–∞: 50"). Triggered by `affix: 'implicit'` in vendor-adapter
17. **Vendor AND mode = true AND** ‚Äî each property is separate AND condition (ALL must match). Legacy buildVendorRegex used OR-within-AND for non-numeric properties. Use OR mode for "any matches" behavior
