## 2. Анализ внешнего аудита — согласия и разногласия

Внешний аудит делает несколько верных наблюдений и несколько спорных рекомендаций. Ниже — обоснованная позиция по каждому пункту.

### 2.1 "Верхнее меню — не топ-3 проблема"

**✅ СОГЛАСЕН.**

Аудит `docs/UI_AUDIT.md` уже установил, что реальные проблемы были в типографике (10px текст, halation от чистого белого) и контрастах — а не в TopNav. Текущий TopNav (iter 64) реализует правильный паттерн:

```
Логотип → Навигация → Discord hint
```

Это не лендинг, а инструмент. Центрирование меню действительно создаст 3 центра внимания (logo left / tabs center / discord right) и ухудшит баланс. Не делаем.

### 2.2 "Слабая визуальная иерархия уровней Префикс → Категория → Аффикс"

**⚠️ ЧАСТИЧНО СОГЛАСЕН.**

Текущая 3-уровневая иерархия уже существует и реализована через `GroupHeader`:

| Уровень | Что это | Текущий стиль | Визуальный вес |
|---------|---------|---------------|----------------|
| **L1** | Префикс / Суффикс / Имплисит | `.affix-header-*` — gradient bg + 4px border-l + corner accents + 16px font-bold | Высокий |
| **L2** | Origin section (Обычные / Осквернённые / Очернённые) | inline-стиль — 14px font-bold + border-l-2 + лёгкий bg-tint | Средний |
| **L3** | Semantic sub-group (ДОБЫЧА / УСИЛЕНИЯ / ...) | GroupHeader variant='sub' — 12px font-semibold + bg + border | Низкий |

Проблема: **L2 слишком слабо отличается от L3.** L1 имеет ярко выраженный "фрейм" (gradient + corner accents), L2 — только border-l-2 и bg-tint, L3 — такой же border + bg, но меньше. Разница между L2 и L3 фактически только в размере шрифта (14 vs 12) и весе (bold vs semibold).

**Решение (iter 164, P1):** Дать L2 собственный визуальный язык — gradient bg + 3px border-l + лёгкие corner accents, но в более "тихом" варианте чем L1. Это создаст чёткий контраст L1 (большой фрейм) → L2 (средний фрейм) → L3 (плоский badge).

### 2.3 "Цветовая система — цвета не несут информации"

**❌ НЕ СОГЛАСЕН с диагнозом. ⚠️ СОГЛАСЕН с симптомом.**

Аудит утверждает: "Синий = защита, Красный = крит, Красный = скорость, Красный = урон, Зелёный = луки, Фиолетовый = ауры, Жёлтый = копья" — и заключает, что цвет перестаёт нести информацию.

Фактически в текущем коде цвет **семантически нагружен** и используется на двух уровнях:

1. **Affix type (primary semantic axis):** blue = prefix, orange = suffix, amber = implicit. Это цвет аффикс-колонки (L1 header) и цвет badge в SelectedBasket. Эта ось фундаментальна — она соответствует игровой механике (префиксы и суффиксы заполняют разные слоты предмета).

2. **Functional category (secondary axis):** цвет bg-tint у L3 sub-group (ДОБЫЧА = зелёный, ЗАЩИТА = синий, УРОН = красный и т.д.). Это информационный маркер внутри категории.

**Симптом "цвета сливаются в шум"** возникает НЕ из-за избытка цветов, а из-за того, что L2 (origin) и L3 (sub-group) имеют похожую интенсивность bg-tint. После усиления L2 (см. §2.2) — контекст станет очевиднее.

**❌ НЕ делаем** предложение аудита "Red = Strength / Green = Dex / Blue = Int" для категорий. Это другая доменная ось (атрибуты персонажа, а не типы модов), и смешивание её с функциональными категориями создаст БОЛЬШЕ путаницы, а не меньше. Текущие affix-type colors (blue/orange/amber) уже устоявшиеся в экосистеме PoE2db/poe2.re.

### 2.4 "Regex блок должен быть визуальным центром правой панели"

**✅ СОГЛАСЕН.**

RegexOutput — это единственный блок, ради которого пользователь вообще открыл инструмент. Сейчас он имеет золотой фрейм (`.regex-output` — gradient + gold border + corner accents), но:

- SelectedBasket прямо над ним визуально соревнуется за внимание.
- StatusPanel под ним имеет похожую структуру (panel + border).
- Активная обратная связь при выборе аффикса — только изменение числа в SelectedBasket.

**Решение (iter 164, P3):** Усилить `.regex-output` — чуть ярче gold border, чуть заметнее glow, добавить subtle pulse-анимацию на момент обновления regex. Это даст пользователю мгновенную обратную связь "выбрал → получил результат".

### 2.5 "Визуальный шум — добавить Compact/Extended режимы"

**⚠️ ЧАСТИЧНО СОГЛАСЕН.**

Текущая система уже имеет partial-noise-reduction через `collapsedGroups` (top-level collapse) и `expandedSubGroups` (sub-group collapse по умолчанию). Это функционально эквивалентно "Compact mode" из аудита.

Что можно добавить позже (НЕ в iter 164): single-toggle "Свернуть все подкатегории" — кнопка, которая сразу добавляет все sub-group keys в `expandedSubGroups = {}`. Это дешёвое дополнение, но оно требует UI-решения (где разместить кнопку), поэтому откладывается.

### 2.6 "Активная вкладка должна выделяться сильнее"

**✅ СОГЛАСЕН.**

Текущий `.nav-mode-active` имеет: gold gradient bg + 3px gold border-b (через `::after`) + 600 font-weight + box-shadow glow. Это decent, но на ярких мониторах gold-on-warm-brown недостаточно контрастен.

**Решение (iter 164, P2):** Чуть усилить — поднять alpha в gradient (0.14 → 0.20), поднять border-b (3px → 3px но с brighter gold), добавить text-shadow для active label. Не ломая существующий паттерн.

### 2.7 "Сделать навигацию цельной панелью вместо отдельных кнопок"

**❌ НЕ СОГЛАСЕН.**

Текущие tab-chips (`--chip-bg` + border + rounded) лучше для touch-targets на мобильных и лучше скроллятся на узких экранах. "Цельная панель" в стиле bootstrap-nav плохо работает при горизонтальном скролле (overflow-x: auto) — становится непонятно, что скроллится.

Не делаем.

## 5. План валидации

После реализации iter 164:

1. **tsc 0 errors** — типы не сломаны.
2. **eslint 0 errors** — стиль кода чистый.
3. **vitest 2319+/2319+ PASS** — все существующие тесты проходят (изменения визуальные, тесты UI не должны сломаться; если сломаются — фиксим).
4. **vite build PASS** — bundle size не должен вырасти заметно (ожидаемо < 1 KB на CSS).
5. **Ручная визуальная проверка пользователем** (30 секунд):
   - Открыть категорию (например, Amulet).
   - L2 (Обычные/Осквернённые) должен иметь явный gradient + border-l, отличный от L3 sub-group.
   - Активная вкладка в TopNav должна быть чуть заметнее.
   - RegexOutput должен иметь лёгкое свечение при изменении regex.
