## Группа A: Логика поиска (AND / OR / NOT)

### A1: AND — два мода на одном предмете
**Гипотеза:** `"X" "Y"` подсвечивает только предметы, где есть ОБА мода.
**Regex:** `"приспешники наносят" "от огня"`

### A2: OR — любой из вариантов
**Гипотеза:** `"X|Y"` подсвечивает предметы с X ИЛИ Y.
**Regex:** `"от огня|от холода"`

### A3: NOT — исключение модов
**Гипотеза:** `"!X|Y"` СКРЫВАЕТ предметы с X или Y.
**Regex:** `"!проклят|сопротивлен"`

### A4: Комбинация хочу + не хочу
**Гипотеза:** Можно совместить "хочу X" AND "не хочу Y".
**Regex:** `"приспешники наносят" "!проклят"`

## Группа G: Дробные числа (H1) — ПРИОРИТЕТ: HIGH

> Источник: Ring 3 "Регенерация 15.9(13.1-18) здоровья в секунду", Amulet 3 "Регенерация 30.9 здоровья"

### G1: Точка в дробном числе — wildcard vs literal
**Гипотеза:** `.` в PoE2 regex матчит любой символ, ВКЛЮЧАЯ литеральную точку в тексте.
**Regex:** `"15.9"`
**Должен:** Ring 3 (Регенерация 15.9 здоровья)
**Наш матчер:** ✅ совпадает (`.` матчит литеральную `.`)

### G2: Без точки — "159" НЕ совпадает с "15.9"
**Regex:** `"159"`
**Должен НЕ совпадать:** Ring 3 (в тексте "15.9", а не "159")
**Наш матчер:** ✅ не совпадает

### G3: Character class для точного совпадения точки
**Regex:** `"15[.]9"`
**Наш матчер:** ✅ совпадает с "15.9"

### G4: Дробные числа и number regex
**Regex:** `"([1-9][0-9]|[0-9][0-9][0-9]).*здоровья в секунду"`
**Наш матчер:** ✅ "15" ≥ 15 → совпадает

**Вывод:** Дробные числа работают корректно. `.` матчит литеральную точку, но также матчит ЛЮБОЙ другой символ. Для точного совпадения точки использовать `[.]`.

## Группа J: Плитки — "зарядов" vs "использ" (H4) — ПРИОРИТЕТ: CRITICAL

> Источник: Заражённая плитка: "Осталось зарядов - 10"

### J1: "зарядов" совпадает с текстом зарядов плитки
**Regex:** `"зарядов"`
**Наш матчер:** ✅ совпадает

### J2: "использ" совпадает с ОПИСАНИЕМ, а не с зарядами!
**Regex:** `"использ"`
**Наш матчер:** ✅ совпадает — НО через "использовать" в описании "Можно использовать в Машине картоходца"
**КРИТИЧЕСКАЯ ПРОБЛЕМА:** "использ" как суффикс для number regex привяжет число к описанию, а не к строке зарядов!

### J3: Число стоит ПОСЛЕ слова "зарядов"
**Текст:** "Осталось зарядов - 10" — число 10 ПОСЛЕ "зарядов"
**Проблема:** Directional `.*` требует число ДО суффикса. "10.*зарядов" не работает!
**Решение:** `"зарядов.*10"` — инвертированный порядок (слово → .* → число)
**Наш матчер:** ✅ `"зарядов.*10"` совпадает

### J4: Cross-mod FP — числа из других модов
**Regex:** `"([1-9][0-9]|[0-9][0-9][0-9]).*зарядов"`
**Наш матчер:** ✅ совпадает — НО через "39" из другого мода (cross-mod FP!)
**Вывод:** Prefix anchoring необходим для фильтрации зарядов.

**ДЕЙСТВИЕ:** Проверить в игре точный текст зарядов плитки. Если "зарядов", а не "использований" — нужно обновить ETL и UI.

## Группа M: `|` внутри `()` и number range с `|` — ВЕРИФИЦИРОВАНО В ИГРЕ ✅

> Источник: генерация number-range regexes в ETL. PoE2 использует `|` для OR и `()` для группировки.
> Предметы для теста: R1 (Отвратительное потрясение), R2 (Расколотый завиток), R3 (Ненавистное потрясение),
> W1-W3 (Путевые камни), T1-T3 (Заражённые плитки), A1-A3 (Лазурные амулеты).

### Предметы и ключевые тексты

| # | Предмет | Ключевые моды |
|---|---------|---------------|
| R1 | Отвратительное потрясение | +66 к макс. здоровья, 28% от огня, +121 к уклонению, +23 к силе, +35% к сопр. молнии |
| R2 | Расколотый завиток | 17–30 урона от огня к атакам, 2–36 урона от молнии к атакам, +32 к силе, +13% к сопр. всем стихиям |
| A1 | Унылый фермуар | 28% реген маны, 43% уклонение, +380 меткость, +12% сопр. холоду, +33 интеллект |
| A2 | Крутящий горжет | 25% реген маны, +184 меткость, +62 ЕС, +24% сопр. молнии |
| A3 | Племенной медальон | 27% реген маны, +17 макс здоровья, +55 маны, +34% сопр. молнии, +14% сопр. хаосу |

### Результаты тестов (in-game, RU клиент)

| Тест | Regex | Ожид. | Факт. | ✅/❌ | Вывод |
|------|-------|:-----:|:-----:|:-----:|-------|
| M-01 | `"огня\|холоду"` | 3 | 3 (2R+1A) | ✅ | `|` без `()` работает — baseline |
| M-02 | `"(огня\|холоду)"` | 3 | 3 (2R+1A) | ✅ | `()` не ломает `\|` — M-02 = M-01 |
| M-03 | `"(огня)"` | 2 | 2 (2R) | ✅ | `()` без `\|` = группировка, не литерал |
| M-04 | `"([3-9][0-9]\|[0-9][0-9][0-9]).*к сопротивлению молнии"` | 2 | 2 (1R+1A) | ✅ | Number range `\|` внутри `()` работает |
| M-05 | `"([0-2][0-9]).*к сопротивлению молнии"` | 1 | 1 (1A) | ✅ | Комплементарный range подтверждает M-04 |
| M-06 | `"(огня\|молнии).*к атакам"` | 1 | 1 (1R) | ✅ | Scoped OR + `.*` работает внутри блока |
| M-07 | `"к сопротивлению" "!молнии\|хаосу"` | 1 | 1 (1A) | ✅ | `!` + `\|` = `!(A\|B)` работает корректно |
| M-08 | `"\d.*к максимуму здоровья"` | 2 | 2 (1R+1A) | ✅ | `\d` поддерживается в PoE2 |

**Пояснения к результатам:**

- **M-07:** Исходное ожидание было 2 предмета (R2 + A1), но R2 содержит «урона от **молнии** к атакам» в другом моде, поэтому `!молнии|хаосу` исключает R2. Фактический результат 1 (только A1) корректен — `!` + `|` работает как `!(A|B)`.
- **M-08:** Исходное ожидание было 3 (R1, R3, A3), но R3 имеет «к макс. **маны**» (мана, не здоровье). Фактические 2 совпадения (R1 с «к макс. здоровья» и A3 с «макс здоровья») подтверждают, что `\d` работает.

### Диагностическое дерево подтверждений

```
M-01 ✅ → базовый | OR работает
M-02 ✅ → () не ломает | (M-02 = M-01)
M-03 ✅ → () = группировка, не литерал
M-04 ✅ → | внутри () с number ranges РАБОТАЕТ
M-05 ✅ → комплементарный range подтверждает M-04 (M-04 ∪ M-05 = все сопр. молнии)
M-06 ✅ → | + .* scoped OR внутри одного блока
M-07 ✅ → ! + | = !(A|B) корректно
M-08 ✅ → \d поддерживается
```

**ИТОГОВЫЙ ВЫВОД:** Все синтаксические конструкции, используемые ETL-компилятором для генерации number-range regex (`|` внутри `()`, `[0-9]` char classes, `\d`), полностью поддерживаются в PoE2 regex движке. Никуда менять `[0-9]` на что-либо другое не нужно — `[0-9]` гарантирует совместимость, а `\d` тоже работает.

## Phase 9: Числовые диапазоны — перечисление (VERIFIED IN-GAME)

### Ключевое открытие

AND двух quoted groups для одного и того же суффикса НЕ работает как числовой диапазон в PoE2. Причина: в тексте предмета с range notation (например `"26(26-50)% шанс откладывания наград"`) вторичное число `"50"` удовлетворяет условию `≥27`, а основное число `"26"` удовлетворяет `≤30`. Каждый quoted group может совпасть с РАЗНЫМ числом в одном блоке.

### Результаты тестов

| Тест | Regex | Результат | Вывод |
|------|-------|-----------|-------|
| 9-1 (OR) | `"карте золота\|предзнаменованиями"` | ✅ Все предметы с хотя бы одним модом подсветились | OR работает корректно |
| 9-2 (AND) | `"откладывания наград" "Обновленные"` | ✅ Только предметы с обоими модами | AND кросс-мод работает корректно |
| 9-3 (AND range) | `"(2[7-9]\|[3-9][0-9]\|[0-9][0-9][0-9]).*откладывания наград" "([0-9]\|[1-2][0-9]\|30).*откладывания наград"` | ❌ FP: 26% и 22% прошли | AND двух quoted groups для одного суффикса — вторичные числа создают FP |
| 9-3a (≥27 only) | `"(2[7-9]\|[3-9][0-9]\|[0-9][0-9][0-9]).*откладывания наград"` | ❌ FP: те же 26% и 22% | Проблема не в AND, а в совпадении вторичных чисел из range notation |

### Решение: Перечисление (enumeration)

Вместо AND двух quoted groups используем единый quoted group с перечислением всех допустимых значений:
```
"(27|28|29|30).*откладывания наград"
```
Это перечисление совпадает ТОЛЬКО с точными значениями 27, 28, 29, 30 — вторичные числа из range notation (50, 47 и т.д.) не совпадают ни с одним из перечисленных значений.

### Реализация

- `generateEnumeratedRangeRegex(min, max)` в `number-regex.ts` генерирует `(27|28|29|30)`
- `MAX_ENUMERATE_RANGE = 50` — порог для переключения на AND fallback
- Диапазоны ≤50 значений → перечисление (точное)
- Диапазоны >50 значений → AND fallback (с известным ограничением)
- `round10` всегда отключён для перечисления — точность встроена в подход
