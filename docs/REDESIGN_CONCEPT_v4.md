## 0. Принципы и методология

### 0.1 Зачем этот документ вместо v3

Документ v3 был «обоснованным анализом + планом» — и сразу перешёл к реализации P1/P2/P3 в той же итерации. Пользователь в iter 164 явно дал понять, что хочет **поэтапного согласования каждого аспекта до изменения кода**. v4 — это попытка пройти тот же путь медленнее: сначала полная карта решений, потом итеративная реализация по одной задаче за раз.

### 0.2 Критерии согласия / несогласия с аудитом

Для каждого пункта аудита я применяю три фильтра:

1. **Факт-чекинг** — что фактически утверждает аудит, и что фактически есть в коде. Многие претензии аудита абстрактны и не опираются на текущий код (аудит писался «по скриншоту», не по коду).
2. **Симптом vs корень** — аудит часто описывает симптом («цвета сливаются в шум») и сразу предлагает решение («убрать все цвета категорий»). Симптом может быть верным, а решение — нет.
3. **Цена изменения** — что ломается, если применить рекомендацию аудита. PoE2 Regex — зрелый проект с 2319 тестами и историями аудита v2/v3; «красивые на бумаге» решения часто ломают семантику, которая складывалась итерациями.

### 0.3 Доменные оси интерфейса

Прежде чем разбирать аудит, важно зафиксировать, что в интерфейсе уже есть **три независимых визуальных оси**, которые нельзя смешивать:

| Ось | Где живёт | Что значит | Цвета |
|-----|-----------|------------|-------|
| **A. Affix type** | L1 header (`affix-header-*`), `SelectedBasket` badge | Игровая механика: prefix/suffix занимают разные слоты предмета | blue / orange / amber |
| **B. Mod origin** | L2 header (`affix-origin-header`), `OriginFilterChip` | Происхождение мода: normal/corrupted/desecrated/essence/breachborn — влияет на способ получения | gray / red / emerald / amber / violet |
| **C. Functional category** | L3 sub-group (`bg-section-*`) | Что мод делает функционально: атака / защита / добыча и т.д. | red / blue / emerald / amber / violet |

**Главное наблюдение:** аудит смешивает оси B и C в одну «цветовую систему» и заключает, что цвет не несёт информации. Это не так. Цвет несут три разные оси, и проблема — в их визуальной силе, а не в количестве.

### Аспект A1. «Слабая визуальная иерархия Префикс → Категория → Аффикс»

> **Цитата аудита:** «визуально различие между уровнями недостаточно выражено. Пользователь вынужден читать структуру, он не может быстро её "увидеть"».

#### Факт-чек

Аудит описывает иерархию из ТРЁХ уровней: Префикс → Категория → Аффикс. В коде фактически **четыре уровня**, и это важное расхождение:

| Уровень | Что это | Где живёт в коде | Текущий визуальный язык |
|---------|---------|------------------|-------------------------|
| **L1** | Affix column (ПРЕФИКСЫ / СУФФИКСЫ / ИМПЛИСЕТ) | `ModList.tsx:407-408` → `.affix-header-{prefix,suffix,implicit}` | gradient bg + 4px border-l + corner accents, 16px bold uppercase |
| **L2** | Origin section (Обычные / Осквернённые / Очернённые / Сущность / Разлом) | `ModList.tsx:446` → `.affix-origin-header` (iter 164) | gradient bg + 3px border-l + small corner accents, 14px bold uppercase + icon |
| **L3** | Functional sub-group (ДОБЫЧА / ЗАЩИТА / УРОН / ...) | `GroupHeader.tsx:91` variant='sub' | flat bg-tint badge, 12px semibold uppercase, no frame |
| **L4** | Аффикс (chip) | `FilterChip.tsx` | flat chip, 13px regular |

Аудит пропустил L2 (origin), потому что не у всех категорий есть origin-subsection (только amulet/ring/belt/jewel с `showOriginSubSections=true`). Waystone/tablet/relic живут в двух уровнях.

#### Валидация диагноза

**ЧАСТИЧНО СОГЛАСЕН.** Диагноз «слабая иерархия» был верен для iter 163 (до iter 164). iter 164 уже усилил L2 (`.affix-origin-header` — gradient + 3px border-l + corner accents), и сейчас разница L1 → L2 → L3 читаема.

Но есть **остаточный дефект**:

- L1 (4px border, 16px bold, large corner accents, opacity 0.4) vs L2 (3px border, 14px bold, small corner accents, opacity 0.35) vs L3 (no frame, 12px semibold) — **разница в каждом параметре есть, но ни один параметр не даёт радикального контраста**.
- На OLED-экранах при яркой засветке L1 и L2 всё ещё читаются как «похожие рамки».

**Корневая причина:** автор решения iter 164 (это я) перестарался с «subtle» — corner accents у L2 opacity 0.35, у L1 opacity 0.4. Разница 0.05 — на грани восприятия.

#### Решение (предлагается для согласования)

**Вариант A — радикальный контраст L2/L3 (рекомендую).** Не трогая L1, сделать так, чтобы L2 и L3 были на разных визуальных «языках»:

- **L2 (origin)** остаётся «фреймовым» (gradient + border + corner accents) — это «контейнер».
- **L3 (functional sub-group)** становится «плоским текстовым маркером» — убираем bg-tint и border, оставляем только цветной текст + маленький разделитель. Это разделит «контейнер» (L2) и «маркер» (L3) по самому типу визуального элемента, а не только по размеру.

**Вариант B — усиление контраста L1/L2 по opacity и толщине.** Менее радикальный:
- L1 corner accents: opacity 0.4 → 0.55, размер 6×6 → 8×8.
- L2 corner accents: opacity 0.35 → 0.30, размер 5×5 → 4×4.
- L1 border-l: 4px, L2 border-l: 3px (как сейчас), L3: 0 (как сейчас).
- Это оставит структуру «фреймовая», но контраст L1/L2 станет ~25% вместо ~12%.

**Вариант C — ничего не трогать.** iter 164 уже улучшил ситуацию; дальнейшее усиление — субъективный вкус.

#### Риски

- Вариант A ломает «паттерн золотой рамки» (`affix-header-*`, `affix-origin-header`, `regex-output` — все используют одну лексику). Удаление bg-tint у L3 может выглядеть как «незаконченность».
- Вариант B сохраняет паттерн, но разница «12% → 25%» может оказаться недостаточной.
- Вариант C закрывает вопрос, но не закрывает ощущение пользователя, что «иерархия слабая».

#### Критерий приёмки (для iter 166, если согласуем)

1. На скриншоте Amulet-страницы при раскрытых L1+L2+L3 пользователь за < 1 секунды отвечает на вопрос «какой это уровень?» для любого блока.
2. Все существующие тесты `tests/ui/GroupHeader.test.tsx`, `tests/ui/ModList.test.tsx`, `tests/ui/VirtualizedModList.test.tsx` — PASS.
3. CSS-размер не вырастает более чем на 0.5 KB.

### Аспект A2. «Цветовая система — цвета не несут информации»

> **Цитата аудита:** «Синий = защита, Красный = крит, Красный = скорость, Красный = урон, Зелёный = луки, Фиолетовый = ауры, Жёлтый = копья. Через несколько минут работы цвет перестаёт что-либо сообщать».

#### Факт-чек

Аудит приводит **вымышленные примеры**. В текущем коде (см. `src/shared/mod-classifier.ts:46-55, 291-301`) фактическое распределение:

| Цвет | Что означает в коде | Примеры из `mod-classifier.ts` |
|------|---------------------|-------------------------------|
| **Красный** | negative / offensive | `offensive` (Атакующие), `negative-monster-power`, `negative-monster-defense`, `ritual` (Ритуал) |
| **Синий** | defensive | `defensive` (Защитные), `delirium-mist` |
| **Зелёный (emerald)** | positive / attribute / loot | `attribute` (Характеристики), `positive-loot` (Добыча), `expedition-*` |
| **Янтарный (amber)** | essence origin / vaal | `essence` (Сущность), `vaal-*` |
| **Фиолетовый (violet)** | breach | `breach-*` (Бездна) |
| **Серый (muted)** | neutral / generic | `neutral-generic`, `generic-loot` |

Аудит утверждает, что «красный = и крит, и скорость, и урон» — **это не так**. Красный последовательно означает «агрессивное/негативное» (offensive + negative + ritual). Зелёный последовательно означает «позитивное/полезное».

#### Валидация диагноза

**НЕ СОГЛАСЕН с диагнозом, но СОГЛАСЕН с симптомом.**

Диагноз аудита неверен — цвета семантически нагружены и консистентны. Но **симптом «цвета сливаются в шум»** реален. Его причины:

1. **L2 (origin) и L3 (functional) используют одну и ту же палитру** (red/blue/emerald/amber/violet). L2 origin «Осквернённые» = красный, и L3 sub-group «Атакующие» = тоже красный. Они находятся рядом в DOM и визуально сливаются.
2. **Насыщенность L3 bg-tint одинаковая** для всех категорий — `bg-section-red`, `bg-section-blue`, `bg-section-emerald` имеют одну и ту же alpha (~0.10-0.15). Глаз не отличает их по «весу».

#### Решение (предлагается для согласования)

**Вариант A — разделить палитры L2 и L3 (рекомендую).**

L2 (origin) использует **происхождение-специфичные** цвета (как сейчас: red=corrupted, emerald=desecrated, amber=essence, violet=breach). Это **B-ось**.

L3 (functional) — переходит на **монохромную** палитру с тонкими цветовыми акцентами только на тексте, без bg-tint. Это убирает конкуренцию между B и C осями. Concretely:
- L3 badge: `bg-panel/15 border-edge/15` (нейтральный) + цветной текст (`text-accent-red`, `text-accent-blue`, ...) для функциональной категории.
- Это позволит L2 сохранять цветной bg-tint (B-ось), а L3 — только цветной текст (C-ось).

**Вариант B — уменьшить насыщенность L3 на 40-60% (по аудиту).**

L3 bg-tint: alpha 0.10-0.15 → 0.05-0.08. Это сделает L3 «приглушённым», и L2 будет доминировать. Минус: L3 потеряет информационную ценность (цвет будет едва виден).

**Вариант C — ничего не делать.**

iter 164 усилил L2, и конкуренция L2/L3 уже уменьшилась. Если после A1 (вариант A) L3 станет монохромным, проблема уходит сама.

#### Риски

- Вариант A ломает визуальный паттерн L3, который задан в `mod-classifier.ts` (там для каждой категории прописан `bgClass` + `borderClass`). Придётся менять схему в коде + во всех 9 категориях.
- Вариант B — визуально дешев, но может выглядеть «грязным» на тёмном фоне.
- Аудиторская идея «Red = Strength / Green = Dex / Blue = Int» — **отвергаем категорически** (см. §3 «Что НЕ делаем»).

#### Критерий приёмки

1. L2 (origin) и L3 (functional sub-group) с одним и тем же цветом (например, оба «красные») на скриншоте читаются как **разные визуальные элементы** (фрейм vs текст-маркер).
2. Все тесты `tests/ui/GroupHeader.test.tsx` — PASS (там проверяется только `variant='sub'` className, не цвет).
3. `mod-classifier.ts` изменения минимальны — только `bgClass`/`borderClass` для sub-group (L3), origin (L2) не трогаем.

### Аспект A3. «Regex блок должен быть визуальным центром правой панели»

> **Цитата аудита:** «По факту это главный результат всей работы пользователя. Но визуально она ощущается второстепенной. `Regex / Профиль / Обозначения` имеют примерно одинаковый визуальный вес».

#### Факт-чек

В коде правая панель (`CategoryLayout.tsx` → `aside`) содержит:
1. `SelectedBasket` (выбранные аффиксы)
2. `RegexOutput` (генерируемый regex)
3. `StatusPanel` (badges + alerts)
4. `ProfilePanel` (опционально, через toggle)
5. `IconLegend` (опционально, через toggle)

`RegexOutput` уже имеет:
- `.regex-output` CSS-класс (gradient bg + gold border-l 3px + box-shadow glow + corner accents).
- iter 164: pulse-анимация на изменение regex (600ms, `prefers-reduced-motion` уважается).
- Health bar (green/yellow/red) + overflow indicator.

`SelectedBasket` — простой `bg-panel border-edge-panel rounded p-3`. `StatusPanel` — такой же. `ProfilePanel` и `IconLegend` — такие же.

**Утверждение аудита «одинаковый визуальный вес» — НЕВЕРНО для iter 164.** RegexOutput уже явно доминирует за счёт золотой рамки и glow. Возможно, аудитор видел старую версию до iter 55, или игнорировал `.regex-output` класс.

#### Валидация диагноза

**ЧАСТИЧНО СОГЛАСЕН.** RegexOutput действительно выделяется, но:

1. **SelectedBasket соревнуется за внимание** в первые секунды, потому что в нём появляются новые чипы (динамика), а RegexOutput — статичен до обновления regex. Pulse-анимация iter 164 частично решает это, но только на момент изменения.
2. **Когда regex пустой** (пользователь ещё ничего не выбрал), RegexOutput визуально пустой и теряет вес — SelectedBasket с placeholder «Выберите аффиксы» тоже пустой, и они выглядят одинаково.

#### Решение (предлагается для согласования)

**Вариант A — визуальная связь SelectedBasket → RegexOutput (рекомендую).**

Добавить тонкий визуальный «поток» от SelectedBasket к RegexOutput:
- Когда в SelectedBasket есть выбранные чипы, между ним и RegexOutput появляется маленький направляющий элемент — стрелка `↓` или тонкая вертикальная линия с золотым градиентом.
- При добавлении чипа в SelectedBasket — стрелка кратко пульсирует (как RegexOutput pulse), давая ощущение «выбрал → поток пошёл → результат готов».
- Когда SelectedBasket пуст — стрелка скрыта.

**Вариант B — пустой RegexOutput placeholder.**

Когда regex пуст, показывать в RegexOutput явный placeholder: тёмная заливка + слабая надпись «Выберите аффиксы, чтобы построить regex» + иконка-стрелка вверх к SelectedBasket. Это усилит «фокус внимания» даже в пустом состоянии.

**Вариант C — оба варианта (A+B).**

Это даст полную связку: пустое состояние → placeholder → выбор → стрелка-поток → результат с pulse.

#### Риски

- Вариант A добавляет JS-логику в `CategoryLayout` (или новый компонент `BasketToRegexFlow`), что усложняет код. Минимально — CSS-only через `:has()` селектор, но поддержка `:has()` в Safari < 15.4 отсутствует (текущая поддержка — 96%+ браузеров, приемлемо).
- Вариант B прост, но placeholder может конфликтовать с уже существующим `t('regex.placeholder')` — нужно его стилизовать, а не заменять.
- Вариант C — больше работы, но даёт лучшее UX.

#### Критерий приёмки

1. При выборе первого аффикса пользователь видит явную реакцию RegexOutput (pulse — уже есть) + явную связь с SelectedBasket (новое).
2. При пустом RegexOutput пользователь не думает «а где результат?» — placeholder подсказывает.
3. Все тесты `tests/ui/RegexOutput.test.tsx` — PASS. Возможно, придётся добавить 1-2 новых теста на placeholder состояние.

### Аспект A4. «Визуальный шум — Compact/Extended режимы»

> **Цитата аудита:** «На экране одновременно присутствуют: меню; фильтры; поиск; категории; аффиксы; regex; профиль; подсказки. Для нового пользователя — перегруз. Добавить два режима: Compact (только категории) / Extended (категории + аффиксы)».

#### Факт-чек

В коде уже есть **два уровня сворачивания** (`docs/UI_REFACTOR_PLAN.md` Phase 2, iter 133):

1. `collapsedGroups: Set<string>` — top-level collapse (сворачивает весь affix-column L1).
2. `expandedSubGroups: Set<string>` — sub-group collapse (L3 по умолчанию свёрнут, разворачивается по клику).

По умолчанию:
- L1 (ПРЕФИКСЫ/СУФФИКСЫ) — развёрнут.
- L2 (origin) — всегда развёрнут (нет collapse-логики).
- L3 (functional sub-group) — **свёрнут по умолчанию**, разворачивается по клику на GroupHeader.

То есть **Compact mode аудита фактически уже существует** — это состояние «L3 свёрнуты». Пользователь видит только L1 + L2 + список L3-заголовков. Extended mode — это когда пользователь раскрыл один или несколько L3.

#### Валидация диагноза

**СОГЛАСЕН с симптомом, НЕ СОГЛАСЕН с решением.**

Симптом реален: на раскрытой категории с 5+ L3 sub-groups экран перегружен. Но:

1. **«Compact/Extended toggle» аудита — это переизобретение existing collapse-логики.** У пользователя уже есть кнопка chevron на каждом L1/L3. Добавление глобального toggle «Свернуть всё / Развернуть всё» — это convenience, а не новый режим.
2. **Глобальный toggle ломает ментальную модель.** Если пользователь раскрыл 2 важных для него L3, а потом нажал «Compact» — он потеряет свой контекст. Это плохо для опыта игрока, который работает с конкретными категориями.

#### Решение (предлагается для согласования)

**Вариант A — кнопка «Свернуть все подкатегории» (рекомендую).**

Добавить в `CategoryControlPanel` (или в панель поиска) маленькую кнопку `⤓ Свернуть подкатегории`. При клике:
- Все L3 sub-groups сворачиваются (`expandedSubGroups` очищается).
- L1 (affix columns) остаются в текущем состоянии.
- L2 (origin) — не трогаем (он и так всегда развёрнут).

Это не «новый режим», а convenience-action — эквивалент «снять все галочки в filter». Не ломает существующее состояние L1.

**Вариант B — кнопка «Развернуть все подкатегории» (дополнение к A).**

Если пользователь хочет посмотреть всё сразу — даём кнопку `⤢ Развернуть`. Это полезно для опытных игроков, которые хотят сделать поиск по всем sub-groups сразу.

**Вариант C — toggle Compact/Extended как у аудита.**

Двухпозиционный switch. Минус: теряется текущее состояние раскрытия при переключении. Не рекомендую.

#### Риски

- Вариант A+B — дешёвый (одна новая кнопка + два action-creator-а в filter-store). Безопасный.
- Вариант C — ломает existing UX, не рекомендую.

#### Критерий приёмки

1. Кнопка «Свернуть подкатегории» появляется только когда есть хотя бы один раскрытый L3.
2. Кнопка «Развернуть подкатегории» появляется только когда есть хотя бы один свёрнутый L3.
3. Действие не влияет на L1 (affix columns) состояние.
4. Все тесты `tests/ui/CategoryLayout.test.tsx`, `tests/store/filter-store.test.ts` — PASS.
